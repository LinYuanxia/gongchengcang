## 一、商品模块概述

### 1.1 模块定位

商品模块是工程材料管理采供一体化平台的核心基础模块，负责管理平台上所有商品的全生命周期，包括商品的创建、审核、上架、价格管理、库存管理、采购和销售等核心功能。商品模块作为双交易链路（工程仓→供应商，施工方→工程仓）的核心载体，连接了供应商、工程仓和施工方三方，是平台业务流转的关键环节。

### 1.2 核心价值

- **统一商品管理**：建立标准化的商品库，确保商品信息的一致性和准确性
- **多角色协同**：支持供应商、工程仓、施工方等多角色在商品生命周期中的协同操作
- **双链路支撑**：为采购和销售两条交易链路提供完整的商品数据和业务支撑
- **库存联动**：商品信息与库存数据实时联动，确保库存准确性
- **价格管控**：支持多层级价格管理，满足不同交易场景的价格需求

### 1.3 模块范围

| 功能分类 | 主要功能 | 涉及角色 |
|---------|---------|---------|
| 商品基础管理 | 商品分类管理、商品属性管理、商品创建、商品编辑 | 平台管理员、供应商 |
| 商品审核流程 | 商品提交审核、商品审核、审核拒绝、重新提交 | 供应商、平台审核员 |
| 商品上下架 | 商品上架、商品下架、商品状态管理 | 供应商、平台管理员 |
| 价格管理 | 供货价格设置、销售价格设置、价格审核、价格调整 | 供应商、工程仓 |
| 库存管理 | 库存查询、库存预警、库存盘点、库存调整 | 工程仓管理员 |
| 商品搜索 | 多条件搜索、筛选、排序、分页 | 所有用户 |
| 采购交易 | 采购单创建、采购单审批、采购单收货 | 工程仓、供应商 |
| 销售交易 | 销售单创建、销售单审批、销售单发货 | 工程仓、施工方 |

### 2.2 核心实体关系说明

| 实体关系 | 关系类型 | 关系说明 |
|---------|---------|---------|
| 商品分类 → 商品SPU | 1:N | 一个分类下可以有多个商品SPU |
| 商品SPU → 商品SKU | 1:N | 一个SPU可以对应多个SKU（不同规格） |
| 商品SPU → 商品属性 | 1:N | 一个SPU可以关联多个属性 |
| 商品属性 → 商品属性值 | 1:N | 一个属性可以有多个属性值 |
| 商品SKU → 供货价格 | 1:N | 一个SKU可以有多个供应商的不同供货价格 |
| 供应商 → 供货价格 | 1:N | 一个供应商可以为多个SKU设置供货价格 |
| 商品SKU → 库存 | 1:1 | 一个SKU在一个工程仓中对应一条库存记录 |
| 工程仓 → 库存 | 1:N | 一个工程仓可以管理多个SKU的库存 |
| 供应商 → 采购订单 | 1:N | 一个供应商可以接收多个采购订单 |
| 工程仓 → 采购订单 | 1:N | 一个工程仓可以创建多个采购订单 |
| 工程仓 → 销售订单 | 1:N | 一个工程仓可以创建多个销售订单 |
| 施工方 → 销售订单 | 1:N | 一个施工方可以创建多个销售订单 |
| 库存 → 库存变动记录 | 1:N | 一条库存记录可以对应多条库存变动记录 |

## 三、标准商品模型设计

### 3.1 商品分类模型 (Category)

#### 核心属性

| 字段名 | 数据类型 | 长度 | 约束 | 描述 |
|-------|---------|------|------|------|
| `id` | `BIGINT` | 20 | `PRIMARY KEY` | 分类ID |
| `parent_id` | `BIGINT` | 20 | `DEFAULT 0` | 父分类ID（0表示顶级分类） |
| `category_name` | `VARCHAR` | 50 | `NOT NULL` | 分类名称 |
| `category_code` | `VARCHAR` | 50 | `UNIQUE NOT NULL` | 分类编码 |
| `category_level` | `TINYINT` | 1 | `NOT NULL` | 分类层级（1-一级分类，2-二级分类，3-三级分类） |
| `icon` | `VARCHAR` | 255 | | 分类图标 |
| `sort` | `INT` | 10 | `DEFAULT 0` | 排序 |
| `status` | `TINYINT` | 1 | `DEFAULT 1` | 状态(1-启用，0-禁用) |
| `create_time` | `DATETIME` | | `DEFAULT CURRENT_TIMESTAMP` | 创建时间 |
| `update_time` | `DATETIME` | | `DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP` | 更新时间 |

#### 业务规则

- **分类层级限制**：最多支持三级分类
- **分类编码规则**：使用层级编码，如 `001` 表示一级分类，`001001` 表示二级分类
- **状态约束**：禁用的分类下不能创建新商品
- **删除约束**：有商品关联的分类不能删除
- **树形结构**：分类采用树形结构组织，支持无限级分类（建议最多三级）

### 3.2 商品属性模型 (Attribute)

#### 核心属性

| 字段名 | 数据类型 | 长度 | 约束 | 描述 |
|-------|---------|------|------|------|
| `id` | `BIGINT` | 20 | `PRIMARY KEY` | 属性ID |
| `category_id` | `BIGINT` | 20 | `FOREIGN KEY` | 分类ID |
| `attr_name` | `VARCHAR` | 50 | `NOT NULL` | 属性名称 |
| `attr_code` | `VARCHAR` | 50 | `UNIQUE NOT NULL` | 属性编码 |
| `input_type` | `TINYINT` | 1 | `DEFAULT 1` | 输入类型(1-文本，2-单选，3-多选，4-下拉选择) |
| `is_required` | `TINYINT` | 1 | `DEFAULT 0` | 是否必填(1-必填，0-选填) |
| `sort` | `INT` | 10 | `DEFAULT 0` | 排序 |
| `status` | `TINYINT` | 1 | `DEFAULT 1` | 状态(1-启用，0-禁用) |
| `create_time` | `DATETIME` | | `DEFAULT CURRENT_TIMESTAMP` | 创建时间 |
| `update_time` | `DATETIME` | | `DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP` | 更新时间 |

#### 业务规则

- **属性分类关联**：属性必须关联到具体的商品分类
- **属性编码唯一**：属性编码在系统中必须唯一
- **输入类型约束**：根据输入类型决定属性值的录入方式
- **必填约束**：必填属性在创建商品时必须填写
- **属性值管理**：属性值可以预设，也可以在创建商品时自定义

### 3.3 商品属性值模型 (AttributeValue)

#### 核心属性

| 字段名 | 数据类型 | 长度 | 约束 | 描述 |
|-------|---------|------|------|------|
| `id` | `BIGINT` | 20 | `PRIMARY KEY` | 属性值ID |
| `attr_id` | `BIGINT` | 20 | `FOREIGN KEY` | 属性ID |
| `value_name` | `VARCHAR` | 100 | `NOT NULL` | 属性值名称 |
| `value_code` | `VARCHAR` | 50 | `UNIQUE NOT NULL` | 属性值编码 |
| `sort` | `INT` | 10 | `DEFAULT 0` | 排序 |
| `create_time` | `DATETIME` | | `DEFAULT CURRENT_TIMESTAMP` | 创建时间 |
| `update_time` | `DATETIME` | | `DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP` | 更新时间 |

#### 业务规则

- **属性值唯一**：同一属性下的属性值名称不能重复
- **编码规则**：属性值编码建议使用 `属性编码_序号` 的形式
- **排序**：属性值按排序字段升序排列

### 3.4 商品SPU模型 (Product)

#### 核心属性

| 字段名 | 数据类型 | 长度 | 约束 | 描述 |
|-------|---------|------|------|------|
| `id` | `BIGINT` | 20 | `PRIMARY KEY` | SPU ID |
| `spu_name` | `VARCHAR` | 200 | `NOT NULL` | 商品名称 |
| `spu_code` | `VARCHAR` | 50 | `UNIQUE NOT NULL` | 商品编码 |
| `category_id` | `BIGINT` | 20 | `FOREIGN KEY` | 分类ID |
| `brand` | `VARCHAR` | 100 | | 品牌 |
| `unit` | `VARCHAR` | 20 | `NOT NULL` | 单位（如：吨、米、个、平方米等） |
| `main_image` | `VARCHAR` | 500 | | 主图URL |
| `images` | `TEXT` | | | 商品图片列表（JSON格式） |
| `description` | `TEXT` | | | 商品描述 |
| `specification` | `TEXT` | | | 规格参数（JSON格式，存储属性键值对） |
| `status` | `TINYINT` | 1 | `DEFAULT 0` | 状态(0-草稿，1-审核中，2-已上架，3-已下架，4-审核拒绝) |
| `supplier_id` | `BIGINT` | 20 | `FOREIGN KEY` | 供应商ID |
| `create_by` | `BIGINT` | 20 | `FOREIGN KEY` | 创建人 |
| `create_time` | `DATETIME` | | `DEFAULT CURRENT_TIMESTAMP` | 创建时间 |
| `update_time` | `DATETIME` | | `DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP` | 更新时间 |

#### 业务规则

- **商品编码规则**：建议使用 `分类编码_日期_序号` 的形式，如 `WL001_20240101_0001`
- **状态流转**：
  ```
  草稿 → 审核中 → 已上架/审核拒绝 → 已下架
                          ↓         ↑
                          └───────┘
  ```
- **供应商关联**：商品必须关联到具体的供应商
- **商品名称唯一**：同一供应商下的商品名称不能重复
- **规格参数**：规格参数以JSON格式存储，便于灵活扩展
- **图片限制**：商品图片最多支持10张，单张图片大小不超过5MB

### 3.5 商品SKU模型 (ProductSKU)

#### 核心属性

| 字段名 | 数据类型 | 长度 | 约束 | 描述 |
|-------|---------|------|------|------|
| `id` | `BIGINT` | 20 | `PRIMARY KEY` | SKU ID |
| `spu_id` | `BIGINT` | 20 | `FOREIGN KEY` | SPU ID |
| `sku_code` | `VARCHAR` | 50 | `UNIQUE NOT NULL` | SKU编码 |
| `sku_name` | `VARCHAR` | 200 | `NOT NULL` | SKU名称 |
| `specs` | `TEXT` | | `NOT NULL` | 规格属性（JSON格式，存储规格键值对） |
| `barcode` | `VARCHAR` | 50 | | 条形码 |
| `weight` | `DECIMAL` | 10,2 | | 重量（kg） |
| `volume` | `DECIMAL` | 10,2 | | 体积（m³） |
| `status` | `TINYINT` | 1 | `DEFAULT 1` | 状态(1-正常，0-禁用) |
| `create_time` | `DATETIME` | | `DEFAULT CURRENT_TIMESTAMP` | 创建时间 |
| `update_time` | `DATETIME` | | `DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP` | 更新时间 |

#### 业务规则

- **SKU编码规则**：建议使用 `SPU编码_序号` 的形式，如 `WL001_20240101_0001_001`
- **规格唯一**：同一SPU下的SKU规格组合必须唯一
- **SKU名称**：SKU名称 = SPU名称 + 规格描述
- **状态约束**：SPU下架时，所有关联的SKU自动下架
- **规格存储**：规格以JSON格式存储，如 `{"颜色":"红色","尺寸":"大号"}`

### 3.6 供货价格模型 (SupplyPrice)

#### 核心属性

| 字段名 | 数据类型 | 长度 | 约束 | 描述 |
|-------|---------|------|------|------|
| `id` | `BIGINT` | 20 | `PRIMARY KEY` | 供货价格ID |
| `supplier_id` | `BIGINT` | 20 | `FOREIGN KEY` | 供应商ID |

| `min_order_quantity` | `INT` | 10 | `DEFAULT 1` | 最小起订量 |
| `max_order_quantity` | `INT` | 10 | | 最大订购量 |
| `delivery_cycle` | `INT` | 10 | | 交货周期（天） |
| `price_type` | `TINYINT` | 1 | `DEFAULT 1` | 价格类型(1-正常价，2-促销价，3-协议价) |
| `effective_date` | `DATE` | | | 生效日期 |
| `expiry_date` | `DATE` | | | 失效日期 |
| `status` | `TINYINT` | 1 | `DEFAULT 1` | 状态(1-有效，0-无效，2-待审核) |
| `create_time` | `DATETIME` | | `DEFAULT CURRENT_TIMESTAMP` | 创建时间 |
| `update_time` | `DATETIME` | | `DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP` | 更新时间 |

#### 业务规则

- **价格唯一性**：同一供应商对同一SKU只能有一个有效的供货价格
- **价格约束**：供货价格必须大于0
- **起订量约束**：最小起订量必须大于0
- **价格类型**：
  - 正常价：默认价格
  - 促销价：促销活动价格，有时间限制
  - 协议价：与特定工程仓的协议价格
- **生效日期**：价格必须在生效日期后才能使用
- **失效日期**：价格超过失效日期后自动失效

  - 商品信息变更审核：商品基本信息变更
- **审核时限**：审核应在24小时内完成
- **拒绝重试**：审核拒绝的商品可以修改后重新提交审核

## 四、商品与供应商、工程仓的关联逻辑

### 4.1 商品与供应商的关联逻辑

#### 4.1.1 关联关系

| 关联类型 | 说明 | 约束 |
|---------|------|------|
| 商品归属 | 商品必须归属于唯一的供应商 | 一个商品只能属于一个供应商 |
| 商品创建 | 供应商只能创建自己的商品 | 供应商不能创建其他供应商的商品 |
| 价格管理 | 供应商管理自己商品的供货价格 | 一个供应商可以为多个SKU设置供货价格 |
| 订单处理 | 供应商处理自己商品的采购订单 | 供应商只能看到自己商品的订单 |

#### 4.1.2 供应商商品管理权限

- **商品创建**：供应商可以创建新商品
- **商品编辑**：供应商可以编辑自己的商品信息
- **商品上下架**：供应商可以申请商品上架和下架
- **价格管理**：供应商可以设置和调整商品的供货价格
- **库存查看**：供应商可以查看自己商品在各工程仓的库存情况
- **订单处理**：供应商可以处理自己商品的采购订单

#### 4.1.3 供应商商品生命周期

```
┌─────────────────────────────────────────────────────────────────────────┐
│                        供应商商品生命周期                                   │
└─────────────────────────────────────────────────────────────────────────┘

供应商创建商品
    ↓
商品草稿状态
    ↓
供应商提交审核
    ↓
商品审核中状态
    ↓
平台审核通过 → 商品已上架 ←→ 供应商管理价格 ←→ 工程仓采购
    ↓                    ↑
平台审核拒绝 → 供应商修改商品信息
    ↓
重新提交审核
```

### 4.2 商品与工程仓的关联逻辑

#### 4.2.1 关联关系

| 关联类型 | 说明 | 约束 |
|---------|------|------|
| 库存管理 | 工程仓管理商品的库存 | 一个SKU在一个工程仓对应一条库存记录 |
| 采购关系 | 工程仓向供应商采购商品 | 一个工程仓可以向多个供应商采购 |
| 销售关系 | 工程仓向施工方销售商品 | 一个工程仓可以向多个施工方销售 |
| 价格使用 | 工程仓使用供应商的供货价格 | 工程仓只能看到供应商设置的供货价格 |

#### 4.2.2 工程仓商品管理权限

- **商品查看**：工程仓可以查看所有已上架的商品
- **商品搜索**：工程仓可以按条件搜索和筛选商品
- **商品收藏**：工程仓可以收藏常用商品
- **采购下单**：工程仓可以向供应商创建采购订单
- **库存管理**：工程仓可以管理自己的商品库存
- **销售定价**：工程仓可以设置商品的销售价格
- **销售接单**：工程仓可以处理施工方的销售订单

#### 4.2.3 工程仓商品生命周期

```
┌─────────────────────────────────────────────────────────────────────────┐
│                        工程仓商品生命周期                                   │
└─────────────────────────────────────────────────────────────────────────┘

工程仓查看商品
    ↓
选择商品创建采购订单
    ↓
供应商接收订单并发货
    ↓
工程仓收货入库 → 库存增加
    ↓
设置销售价格
    ↓
施工方查看并下单 → 工程仓接收订单
    ↓
工程仓发货出库 → 库存减少
    ↓
库存不足 → 触发库存预警 → 重新采购
```

### 4.3 供应商与工程仓的协同逻辑

#### 4.3.1 商品信息协同

- **商品信息同步**：供应商创建的商品信息自动同步到所有工程仓
- **商品状态同步**：商品上下架状态实时同步到所有工程仓
- **价格信息同步**：供应商设置的供货价格实时同步到所有工程仓
- **库存信息同步**：工程仓的库存信息实时同步给对应的供应商

#### 4.3.2 订单协同

- **采购订单流转**：工程仓创建采购订单 → 供应商接收订单 → 供应商发货 → 工程仓收货
- **订单状态同步**：订单状态实时同步给双方
- **订单通知**：订单状态变更时实时通知双方
- **订单追踪**：双方可以实时追踪订单进度

#### 4.3.3 库存协同

- **库存预警**：工程仓库存低于预警值时，自动通知对应的供应商
- **库存查询**：供应商可以查看自己商品在各工程仓的库存情况
- **补货建议**：系统根据库存情况和历史采购数据自动生成补货建议
- **库存调整**：工程仓库存调整后实时同步给供应商

## 五、双链路交易的商品流转逻辑

### 5.1 双交易链路架构

```
┌─────────────────────────────────────────────────────────────────────────┐
│                          双交易链路架构                                   │
└─────────────────────────────────────────────────────────────────────────┘

链路一：工程仓采购链路（B2B采购）
══════════════════════════════════════════════════════════════════════════

    ┌──────────┐                    ┌──────────┐
    │  工程仓   │ ─────采购订单────▶ │  供应商   │
    │(Warehouse)│                    │(Supplier)│
    │           │ ◀─────发货─────────│          │
    │   买方    │                    │   卖方   │
    └──────────┘                    └──────────┘
          │                              │
          │ 1. 创建采购订单               │ 2. 接收订单
          │ 3. 收货入库                  │ 4. 发货
          ▼                              ▼
    ┌─────────────────────────────────────────────────────┐
    │                  商品库存增加 (工程仓)                │
    └─────────────────────────────────────────────────────┘


链路二：施工方采购链路（内部销售）
══════════════════════════════════════════════════════════════════════════

    ┌──────────┐                    ┌──────────┐
    │  施工方   │ ─────采购订单────▶ │  工程仓   │
    │(Construction)│                 │(Warehouse)│
    │           │ ◀─────发货─────────│          │
    │   买方    │                    │   卖方   │
    └──────────┘                    └──────────┘
          │                              │
          │ 1. 创建采购订单               │ 2. 接收订单
          │ 3. 收货确认                  │ 4. 发货
          ▼                              ▼
    ┌─────────────────────────────────────────────────────┐
    │                  商品库存减少 (工程仓)                │
    └─────────────────────────────────────────────────────┘
```

### 5.2 链路一：工程仓→供应商采购链路

#### 5.2.1 采购流程

| 阶段 | 操作 | 说明 | 涉及角色 |
|-----|------|------|---------|
| 1. 商品选择 | 工程仓浏览商品，选择需要采购的商品 | 工程仓可以按分类、品牌、供应商等条件筛选商品 | 工程仓 |
| 2. 创建采购单 | 工程仓创建采购订单，选择商品、填写数量、选择供应商 | 系统自动显示该商品的可用供应商和对应的供货价格 | 工程仓 |
| 3. 采购单审核 | 工程仓内部审核采购订单（可选） | 根据工程仓的审批流程配置，可以设置多级审核 | 工程仓管理员 |
| 4. 供应商接单 | 供应商接收采购订单，确认可以供货 | 供应商可以选择接单或拒绝接单 | 供应商 |
| 5. 供应商发货 | 供应商按照订单要求发货，填写发货信息 | 供应商需要填写物流公司、物流单号等信息 | 供应商 |
| 6. 工程仓收货 | 工程仓收到货物后，确认收货 | 工程仓可以部分收货或全部收货 | 工程仓 |
| 7. 入库完成 | 系统自动增加工程仓的商品库存 | 库存数量 = 原库存 + 收货数量 | 系统自动 |

#### 5.2.2 采购订单状态流转

```
┌─────────────────────────────────────────────────────────────────────────┐
│                        采购订单状态流转                                   │
└─────────────────────────────────────────────────────────────────────────┘

创建订单 → 待审核 → 已审核 → 待供应商接单
                                     ↓
                            供应商接单 → 待发货
                                  ↓
                            供应商发货 → 待收货
                                  ↓
                            工程仓收货 → 已完成
                                  ↓
                              （库存增加）

                ↓ 拒绝接单 ↓ 取消订单 ↓ 拒绝收货
                └───────────────── 已取消 ─────────────────┘
```

#### 5.2.3 采购对商品库存的影响

| 操作 | 库存变化 | 说明 |
|-----|---------|------|
| 创建采购订单 | 无变化 | 库存锁定（可选） |
| 供应商接单 | 无变化 | 确认供货 |
| 供应商发货 | 无变化 | 等待收货 |
| 工程仓收货 | 库存增加 | 实际入库数量 = 收货数量 |
| 取消订单 | 无变化 | 解除库存锁定（如有） |

### 5.3 链路二：施工方→工程仓采购链路

#### 5.3.1 销售流程

| 阶段 | 操作 | 说明 | 涉及角色 |
|-----|------|------|---------|
| 1. 商品选择 | 施工方浏览商品，选择需要采购的商品 | 施工方只能看到已上架且有库存的商品 | 施工方 |
| 2. 创建销售单 | 施工方创建销售订单，选择商品、填写数量、选择工程仓 | 系统自动显示该商品的可用工程仓和对应的销售价格 | 施工方 |
| 3. 销售单审核 | 施工方内部审核销售订单（可选） | 根据施工方的审批流程配置，可以设置多级审核 | 施工方管理员 |
| 4. 工程仓接单 | 工程仓接收销售订单，确认可以供货 | 工程仓需要检查库存是否充足 | 工程仓 |
| 5. 工程仓发货 | 工程仓按照订单要求发货，填写发货信息 | 工程仓需要填写物流公司、物流单号等信息 | 工程仓 |
| 6. 施工方收货 | 施工方收到货物后，确认收货 | 施工方可以部分收货或全部收货 | 施工方 |
| 7. 出库完成 | 系统自动减少工程仓的商品库存 | 库存数量 = 原库存 - 发货数量 | 系统自动 |

#### 5.3.2 销售订单状态流转

```
┌─────────────────────────────────────────────────────────────────────────┐
│                        销售订单状态流转                                   │
└─────────────────────────────────────────────────────────────────────────┘

创建订单 → 待审核 → 已审核 → 待工程仓接单
                                     ↓
                            工程仓接单 → 待发货
                                  ↓
                            工程仓发货 → 待收货
                                  ↓
                            施工方收货 → 已完成
                                  ↓
                              （库存减少）

                ↓ 拒绝接单 ↓ 取消订单 ↓ 拒绝收货
                └───────────────── 已取消 ─────────────────┘
```

#### 5.3.3 销售对商品库存的影响

| 操作 | 库存变化 | 说明 |
|-----|---------|------|
| 创建销售订单 | 库存锁定 | 锁定数量 = 订单数量 |
| 工程仓接单 | 无变化 | 确认供货 |
| 工程仓发货 | 库存减少 | 实际出库数量 = 发货数量 |
| 施工方收货 | 无变化 | 确认收货 |
| 取消订单 | 库存解锁 | 解除库存锁定 |

### 5.4 双链路商品流转的联动逻辑

#### 5.4.1 库存联动

- **采购入库**：工程仓向供应商采购，库存增加
- **销售出库**：施工方向工程仓采购，库存减少
- **库存预警**：当库存低于预警值时，系统自动触发预警，提醒工程仓补货
- **库存锁定**：销售订单创建后自动锁定库存，避免超卖
- **库存同步**：库存变动实时同步给供应商和施工方

#### 5.4.2 价格联动

- **供货价格**：供应商设置供货价格，工程仓采购时使用
- **销售价格**：工程仓基于供货价格设置销售价格，施工方采购时使用
- **价格调整**：供应商调整供货价格后，工程仓可以同步调整销售价格
- **价格审核**：供货价格和销售价格的调整都需要经过审核
- **价格生效**：价格调整审核通过后自动生效

#### 5.4.3 订单联动

- **采购订单→销售订单**：工程仓的采购订单可以直接转换为销售订单（用于快速补货）
- **销售订单→采购订单**：销售订单库存不足时，可以自动生成采购订单建议
- **订单状态同步**：采购订单和销售订单的状态实时同步给相关方
- **订单提醒**：订单状态变更时自动发送提醒通知

#### 5.4.4 数据联动

- **库存数据**：库存数据实时同步，供应商和工程仓都可以查看
- **销售数据**：销售数据实时统计，供应商可以查看自己商品的销售情况
- **采购数据**：采购数据实时统计，工程仓可以查看自己的采购情况
- **数据分析**：系统基于历史数据提供采购和销售建议

## 八、商品模块业务规则汇总

### 8.1 商品创建规则

1. **商品编码规则**：建议使用 `分类编码_日期_序号` 的形式，确保唯一性
2. **商品名称规则**：同一供应商下的商品名称不能重复
3. **商品分类规则**：商品必须关联到具体的分类
4. **商品图片规则**：商品图片最多支持10张，单张图片大小不超过5MB
5. **商品规格规则**：规格参数以JSON格式存储，便于灵活扩展

### 8.2 商品审核规则

1. **审核流程**：商品必须经过审核才能上架
2. **审核时限**：审核应在24小时内完成
3. **审核拒绝规则**：审核拒绝的商品可以修改后重新提交审核
4. **审核记录规则**：所有审核操作都需要记录审核人和审核时间

### 8.3 价格管理规则

1. **价格唯一性规则**：同一供应商对同一SKU只能有一个有效的供货价格
2. **价格约束规则**：价格必须大于0
3. **价格生效规则**：价格必须在生效日期后才能使用
4. **价格失效规则**：价格超过失效日期后自动失效
5. **价格调整规则**：价格调整需要经过审核

### 8.4 库存管理规则

1. **库存唯一性规则**：一个SKU在一个工程仓中对应一条库存记录
2. **库存数量规则**：库存数量不能为负数
3. **库存锁定规则**：销售订单创建后自动锁定库存，避免超卖
4. **库存预警规则**：当库存低于预警值时，系统自动触发预警
5. **库存变动规则**：所有库存变动都需要生成库存流水记录

### 8.5 采购订单规则

1. **订单创建规则**：采购订单必须关联到具体的供应商和商品
2. **订单数量规则**：订单数量必须大于0
3. **订单状态流转规则**：订单状态必须按照规定的流程流转
4. **订单取消规则**：已发货的订单不能取消
5. **订单收货规则**：可以部分收货或全部收货

### 8.6 销售订单规则

1. **订单创建规则**：销售订单必须关联到具体的工程仓和商品
2. **订单数量规则**：订单数量必须大于0且不超过可用库存
3. **订单状态流转规则**：订单状态必须按照规定的流程流转
4. **订单取消规则**：已发货的订单不能取消
5. **订单收货规则**：可以部分收货或全部收货
