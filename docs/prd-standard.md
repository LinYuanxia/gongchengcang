# 工程仓 PRD 标准规范 V2.0

---

## 一、设计理念

### 1.1 核心目标
**零沟通成本开发** - 研发拿到 PRD 后不需要问产品任何问题，即可 100% 准确实现所有功能。

### 1.2 三大黄金标准

| 标准 | 说明 | 反模式 |
|------|------|--------|
| **字段级细化** | 每个输入框都要有：类型、必填、长度、验证规则、占位提示 | "做一个表单" |
| **全链路流程图** | 每个交互都要有完整流程：用户操作 → 前端校验 → 后端校验 → 成功/失败反馈 | "点击按钮提交" |
| **权限矩阵** | 每个按钮都要明确：哪些角色能看，哪些角色不能看 | "管理员能操作" |

## 二、Markdown 扩展语法大全

### 2.1 标题层级

| 语法 | 渲染效果 | 使用场景 |
|------|---------|----------|
| `### 标题内容` | 带 📌 图标 + 渐变橙白背景 + 左侧 5px 粗橙边 | 一级章节 |
| `#### 标题内容` | 左侧 4px 黄线 + 底部渐变下划线 | 二级小节 |

### 2.2 列表组件

**语法：**
```markdown
- 列表内容自动变成卡片
```

**渲染效果：**
- ✅ 勾选图标
- 浅灰色背景卡片
- Hover 右移 4px + 橙色边框 + 阴影

### 2.3 表格组件

**标准 Markdown 表格：**
```markdown
| 字段名 | 组件 | 必填 | 长度限制 |
|--------|------|------|----------|
| 商户名称 | Input | ✅ 是 | 最多 128 字符 |
```

**渲染效果：**
- 橙色渐变表头
- 斑马纹行背景
- Hover 行橙色高亮
- 圆角 12px + 卡片阴影

### 2.4 流程图组件

**语法：**
```markdown
>>>flow
点击添加 -> 打开抽屉 -> 前端校验 -> 提交后端
成功 -> 关闭抽屉 -> 刷新列表 -> Message提示
失败 -> 回滚状态 -> 错误提示 -> 保留数据
<<<
```

**渲染效果：**
- 蓝紫渐变背景
- 白色卡片节点
- 白色箭头
- 「🔄 业务流程」左上角标签

### 2.5 提示框组件

| 语法 | 类型 | 图标 | 颜色 |
|------|------|------|------|
| `>> 提示内容` | 普通提示 | 💡 | 蓝色左边框 |
| `!! 警告内容` | 警告提示 | ⚠️ | 橙色左边框 |

### 2.6 标签组件

| 语法 | 渲染效果 |
|------|---------|
| `[[标签文字]](primary)` | 紫色渐变标签 |
| `[[标签文字]](success)` | 绿色渐变标签 |
| `[[标签文字]](warning)` | 橙色渐变标签 |

### 2.7 分割线

**语法：** `---`

**渲染效果：** 渐变三色分割线

## 三、内容标准 - 强制执行

### 3.1 列表页 PRD 七件套

每个列表页必须包含以下 7 个模块，缺一不可：

| 模块 | 必须包含的内容 |
|------|-------------|
| **1. 搜索筛选模块** | 每个筛选器的：宽度、类型、选项值、搜索范围、联动逻辑 |
| **2. 表格列定义** | 每一列的：宽度、对齐方式、特殊渲染逻辑 |
| **3. Tag 色彩规范** | 每个状态值对应的颜色、样式（必须精确） |
| **4. 行操作按钮** | 每个按钮的：权限、前置条件、点击后的完整交互 |
| **5. 状态机流转图** | 所有状态的完整流转 + 前置条件 + 触发动作 |
| **6. 权限矩阵** | 4 角色 × N 功能点 的完整对照表（能看/不能看） |
| **7. 空数据规范** | 各种异常场景的处理方式（null/undefined/空字符串） |

### 3.2 表单页 PRD 五件套

每个表单页必须包含以下 5 个模块，缺一不可：

| 模块 | 必须包含的内容 |
|------|-------------|
| **1. 入口规范** | 按钮位置、样式、文字、权限、前置条件 |
| **2. 完整交互流程图** | 从点击按钮 → 最终结果的全链路 |
| **3. 字段级定义表** | 每个字段 6 维度：字段名、组件、必填、长度、验证规则、占位提示 |
| **4. 分步校验规则** | 每一步的校验时机、失败处理、阻断逻辑 |
| **5. 成功/失败反馈** | 各种场景的用户反馈（关闭/提示/保留数据） |

## 四、字段级定义 6 维度标准模板

### 4.1 模板定义

```markdown
| 字段名 | 组件类型 | 必填 | 长度限制 | 验证规则 | 占位提示 |
|--------|---------|------|----------|----------|----------|
| 商户名称 | Input输入 | ✅ 是 | 最多 128 字符 | 非空 | 请输入商户名称 |
| 联系人手机号 | Input输入 | ✅ 是 | 最多 20 字符 | 匹配 ^1[3-9]\d{9}$ | 请输入联系人手机号 |
| 统一社会信用代码 | Input输入 | ✅ 是 | 固定 18 位 | 长度 = 18 | 请输入18位统一社会信用代码 |
```

### 4.2 常用组件类型清单

- Input 单行文本
- Textarea 多行文本
- Select 下拉单选
- Radio 单选组
- Checkbox 多选框
- DatePicker 日期选择
- Upload 图片上传
- InputNumber 数字输入
- Switch 开关

### 4.3 必填标记规范

- ✅ 是 - 红色星号标记，前端必校验
- ❌ 否 - 无星号，用户可留空
- * 条件必填 - 备注说明条件

## 五、权限矩阵标准模板

### 5.1 标准四角色定义

| 角色 | 说明 |
|------|------|
| 👑 超级管理员 | 平台最高权限 |
| 👔 平台管理员 | 全功能操作权限 |
| 🧑‍💼 运营人员 | 日常运营权限 |
| 👁️ 只读用户 | 仅查看权限 |

### 5.2 标准对照表模板

```markdown
| 功能点 | 超管 | 管理员 | 运营 | 只读 |
|--------|------|--------|------|------|
| 查看列表 | ✅ | ✅ | ✅ | ✅ |
| 新增数据 | ✅ | ✅ | ✅ | ❌ |
| 编辑数据 | ✅ | ✅ | ✅ | ❌ |
| 删除数据 | ✅ | ✅ | ❌ | ❌ |
| 进入系统 | ✅ | ❌ | ❌ | ❌ |
```

### 5.3 无权限处理原则

**核心原则：看不见 = 不存在**

| 层级 | 正确方式 | 错误方式（禁止） |
|------|---------|----------------|
| 按钮级 | 直接不渲染 | 按钮灰掉禁用 |
| 页面级 | 403 无权限页 | Toast 提示"你没有权限" |
| 接口级 | 返回 403 HTTP 状态 | code 500 + 报错信息 |

## 六、用户交互标准

### 6.1 加载状态
- 所有异步操作按钮必须显示 Loading
- 接口请求期间按钮不可重复点击
- 表格请求期间显示骨架屏或加载动画

### 6.2 成功反馈
- Message 提示：绿色、3秒自动消失
- 操作成功后自动刷新列表
- 新增/编辑成功自动关闭弹窗/抽屉

### 6.3 失败反馈
- Message 提示：红色、5秒停留或手动关闭
- 表单状态回滚（如 Switch 拨动失败要回滚）
- 用户输入的数据必须 100% 保留，绝不清空

### 6.4 二次确认场景

以下操作必须弹出二次确认框：
- 删除操作
- 停用/启用
- 状态流转（审核通过/驳回）
- 批量操作

## 八、验收标准

一份合格的 PRD，研发看到后应该产生以下三种反应：

1. ✅ **"哦，这么详细啊"** - 超出预期的细致程度
2. ✅ **"这个我知道怎么做了"** - 没有模糊不清的地方
3. ✅ **"不用再问产品了"** - 所有边界场景都覆盖到了

如果研发还在问以下问题，说明 PRD 不合格：

- ❌ "这个字段最长多少？"
- ❌ "这个按钮谁能点？"
- ❌ "失败了怎么提示？"
- ❌ "点完之后呢？"
