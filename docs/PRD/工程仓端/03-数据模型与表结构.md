## 一、功能概述

### 1.1 功能定位

本文档定义工程仓端所有业务数据的**表结构、字段规范、实体关系**，是后端开发和数据库设计的核心参考。涵盖商户、员工、订单、库存、入库、出库、购物车、采购计划、售后、财务等18张核心表。

### 1.2 核心概念

| 概念 | 说明 | 涉及表 |
|-----|------|--------|
| 工程仓主体 | 工程仓的企业信息 | warehouse_merchant |
| 子仓库 | 工程仓下的独立物理仓库 | sub_warehouse |
| 三状态订单 | 同时维护订单/支付/发货三个独立状态的订单表 | orders |
| 采购扩展 | 采购订单特有的供应商和仓库信息 | purchase_order |
| 销售扩展 | 销售订单特有的施工方和物流信息 | sale_order |
| 库存 | 按仓库+SKU维度的库存数量 | inventory |
| 入库批次 | 每次收货的详细记录 | inbound_batch |
| 乐观锁 | 通过version字段控制并发 | inventory |

### 1.3 核心设计原则

1. **三状态分离**：orders表同时维护order_status/payment_status/ship_status三个独立字段
2. **乐观锁并发**：inventory表使用version字段防止超卖
3. **采购/销售隔离**：purchase_order和sale_order通过orders表关联，各自扩展不同字段
4. **批次追溯**：入库操作生成批次记录，支持全链路溯源

## 二、业务数据规则

### 2.1 字段命名规范

- 所有表使用BIGINT自增主键（id）
- 创建时间统一字段：create_time（DATETIME, DEFAULT NOW）
- 更新时间统一字段：update_time（DATETIME, ON UPDATE）
- 状态字段统一使用VARCHAR(20)可读字符串，不使用数字枚举
- 金额字段统一使用DECIMAL(18,2)
- 外键字段统一使用BIGINT，命名规则：关联表名_id

### 2.2 订单数据规则

- 订单编号格式：PO+年月日+序号（采购）/ SO+年月日+序号（销售）
- 三状态字段不可为NULL，都有默认值
- 一条订单记录对应多个order_items
- purchase_order和sale_order为orders的子表（1:1关系）
- 订单金额=所有明细小计之和

### 2.3 库存数据规则

- 库存唯一索引：(warehouse_id, sku_id)，一个仓库一个SKU一条记录
- 库存更新必须在事务中执行
- 出库时使用乐观锁：`UPDATE ... SET quantity=quantity-?, version=version+1 WHERE version=?`
- 库存不能为负数（约束：quantity >= 0）

### 2.4 批次数据规则

- 每批入库生成独立批次号（系统自动）
- 批次号格式：BAT+年月日+序号
- 批次关联入库单ID和SKU ID
- 批次记录实收数量和货损数量

## 四、实体关系总结

| 实体 | 分类 | 核心关联 |
|:----:|:----:|---------|
| warehouse_merchant | 商户 | → employee, sub_warehouse |
| sub_warehouse | 仓库 | → inventory, inbound_order, outbound_order, stock_check |
| employee | 用户 | → 角色, 订单操作 |
| orders | 订单 ⭐ | → order_items, purchase_order, sale_order, after_sale, fund_flow |
| purchase_order | 采购扩展 | → supplier, warehouse |
| sale_order | 销售扩展 | → construction |
| inventory | 库存 ⭐ | → sub_warehouse, sku |
| inbound_order | 入库 | → order, inbound_batch |
| outbound_order | 出库 | → order, outbound_item |
| cart_item | 购物车 | → employee, sku, supplier |
| purchase_plan | 采购计划 | → plan_item |
| after_sale | 售后 | → orders |
