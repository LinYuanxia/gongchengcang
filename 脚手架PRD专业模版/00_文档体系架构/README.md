# PRD 文档脚手架 — 规范说明与复盘分析

> 基于「工程仓端 SaaS」实际 PRD 文档结构提炼的可复用脚手架模板
> 版本：v1.0 | 复盘日期：2026-04-24

---

## 一、复盘总结

### 1.1 文档体系设计演进

本脚手架源自对一个完整 SaaS 工程仓端 PRD 文档库（15 个 MD 文件、770+ 行总纲）的实际编写过程复盘。在多次迭代中，逐步形成了以下核心认知：

| 迭代阶段 | 核心动作 | 关键发现 |
|---------|---------|---------|
| 阶段一：独立文档 | 按模块分别撰写功能设计 | 文档间缺乏关联，读者无法形成全局认知 |
| 阶段二：追加总纲 | 创建 prd.md 统一索引 + 设计原则 | 全局视角建立，但各独立文档格式不统一 |
| 阶段三：格式标准化 | 每篇文档统一头部元数据、功能概述、业务规则 | 可读性提升，但缺少状态治理视角 |
| 阶段四：补充矩阵 | 追加状态×操作矩阵、角色权限表、错误提示汇总 | 文档从"功能描述"升级为"完整规格说明书" |
| 阶段五：复盘脚手架 | 提炼通用模板，形成可复用的 PRD 编写规范 | **即本脚手架** |

### 1.2 文档结构「六层模型」

所有 PRD 文档应遵循以下六层结构，从宏观到微观逐步收敛：

```
┌─────────────────────────────────────────────────────┐
│  Layer 1：文档头部元数据                              │
│  版本/状态/日期/所属章节/负责人 — 每篇文档必有          │
├─────────────────────────────────────────────────────┤
│  Layer 2：功能概述                                    │
│  功能定位 / 核心概念(表) / 目标用户 / 模块范围(表)      │
├─────────────────────────────────────────────────────┤
│  Layer 3：业务规则                                    │
│  状态规则 / 校验规则 / 查询规则 / 命名规范 — 可读文字   │
├─────────────────────────────────────────────────────┤
│  Layer 4：功能点详细设计（核心主体）                    │
│  逐个功能点的 说明/入口/交互/字段/规则/UI/异常/权限     │
├─────────────────────────────────────────────────────┤
│  Layer 5：状态治理矩阵                                │
│  动作定义表 / 状态×操作矩阵 / 每状态角色权限 / 错误提示  │
├─────────────────────────────────────────────────────┤
│  Layer 6：附录                                        │
│  Mermaid 流程图 / 版本历史                            │
└─────────────────────────────────────────────────────┘
```

### 1.3 关键规范要点

| 规范项 | 要求 | 说明 |
|-------|------|------|
| 文档命名 | `NN-文档名称.md` | 两位数序号（01-99），名称简洁 |
| 一级标题 | `# 项目名 - 文档类型` | 统一格式，不加编号 |
| 头部元数据 | 版本/状态/所属章节 | 每篇文档必须有，blockquote 格式 |
| 标题层级 | 一级 `#` → 二级 `##` → 三级 `###` → 四级 `####` | 最多四级 |
| 编号体系 | 一 → 二 → 三 ... / 1.1 → 1.2 ... | 全文统一编号风格 |
| 表格 | 第一行=表头，第二行=分隔符 | 复杂表用 Markdown 标准表格 |
| 代码块 | 指定语言（typescript/sql/mermaid） | 规范语法高亮 |
| 流程图 | `mermaid` 代码块 | graph TB 方向，节点文字清晰 |
| 状态×操作矩阵 | ✅ / ❌ / →说明 | 统一符号，不可用空单元格 |
| 功能点编号 | 模块内唯一（如 PUR-01） | 前缀取模块英文缩写 |
| 文档索引 | prd.md 的「零、文档索引」章节 | 所有文档需在总纲中登记 |

## 二、脚手架文件清单

```
prd-脚手架/
├── README.md                          ← 本文档：复盘分析 + 规范总览
├── prd.md                             ← 文档总纲模板
├── 01-系统概览与架构.md               ← 系统级文档模板
├── 02-业务流程设计.md                 ← 流程设计模板
├── 04-领域模型设计.md                 ← 领域模型模板
└── 05-系统模块详细设计.md             ← 模块功能详细设计模板（最常用）
```

### 各文件说明

| 模板文件 | 对应场景 | 使用频率 | 预计页数 |
|---------|---------|:--------:|:--------:|
| `prd.md` | 任何 PRD 项目起步，创建文档总纲 | 每个项目 1 篇 | 长文（跨模块视角） |
| `01-系统概览与架构.md` | 新系统/子系统介绍 | 每个项目 1 篇 | 中篇 |
| `02-业务流程设计.md` | 跨模块的核心业务流程 | 每个项目 1 篇 | 长篇（含多张流程图） |
| `04-领域模型设计.md` | 后端 DDD 实现参考 | 每个项目 1 篇 | 中篇 |
| `05-系统模块详细设计.md` | **每个功能模块各 1 篇** | 每模块 1 篇 | 长篇（核心产出） |

## 三、文档编写总则

### 3.1 语言风格
- 中文撰写，专业名词保留英文（SKU/SPU/BOM/DDL/RBAC）
- 用词统一：全文保持术语一致（不要混用"用户"/"账号"/"员工"）
- 语气客观，避免主观评价（不写"很复杂"/"很简单"）

### 3.2 表格规范
- 表头首行，第二行分隔符 `|:---|:---|:---:|`
- 单元格内容简洁，超过 20 字考虑换行
- 权限表统一符号：✅ 有权限 / ❌ 无权限 / ➡ 特殊说明 / — 不适用

### 3.3 代码块规范
- 业务逻辑伪代码/接口定义 → `typescript`
- 数据库 DDL → `sql`
- 流程图/状态图 → `mermaid`
- 配置/数据示例 → `json`

### 3.4 Mermaid 流程图规范
- 标题节点用 subgraph 分组
- 避免孤立节点（每个节点必须有入边或出边）
- 使用 style 定义颜色区分不同状态
- 状态节点命名 `${状态英文}`（如 `pending[待确认]`）
- 每条边标注触发条件：`-->|操作名称|`

### 3.5 状态治理矩阵规范
- 动作定义表：每个功能点分配唯一动作ID
- 状态×操作矩阵：行为维度的交叉分析
- 每状态角色权限表：用户维度的二次分析
- 错误提示汇总：提前穷举异常场景，避免开发中遗漏

## 四、使用说明

1. 将 `prd-脚手架/` 目录复制到新项目
2. 将 `工程仓端` 替换为实际项目名称
3. 按「文档总纲 → 系统概览 → 流程设计 → 数据模型 → 领域模型 → 模块详细设计」顺序编写
4. 每完成一个模块的详细设计，同步更新 prd.md 的文档索引
5. 最后补全 Mermaid 流程图和版本历史
